2. 这个 Agent 到底是什么
  2.1 它不是普通聊天总结器
    - 它是项目里的第一个“深度分析 Agent”
    - 专门处理“搬史 / 吃史 / QQ 群史文化”这类目标
  2.2 它的核心目标
    - 识别这是不是史
    - 识别这是什么类型的史
    - 解释它为什么是史
    - 识别搬运 / 转运 / 二手套娃结构
    - 在有需要时，用贴近群友语感的方式说出来
    - 进一步的话，能接这种史的下话茬

3. 为什么不能直接做成一个“会说抽象话的大模型”
  3.1 因为“懂史”和“会整活”不是一回事
    - 会说怪话，不代表真的懂这坨史的结构
    - 会接梗，不代表真的知道为什么这是史
  3.2 所以必须拆层
    - 证据层
    - 判断层
    - 文化解释层
    - 口吻渲染层
    - 接茬能力层

4. 这个 Agent 内部准备怎么拆
  4.1 Stage A: 证据综合
    - 看文本
    - 看 reply
    - 看 forward / nested forward
    - 看 recurrence / 重复搬运
    - 看 share/system marker
    - 看多模态摘要
    - 看缺失媒体造成的未知区
  4.2 Stage B: 史判断
    - 这段里有没有史
    - 属于什么类型
    - 是高价值史还是工业废史
    - 是原生还是外源，还是二手套娃
  4.3 Stage C: 文化解释
    - 为什么它成立
    - 荒诞感在哪
    - 错位感在哪
    - 包浆感在哪
    - 群体共振点在哪
  4.4 Stage D: 口吻渲染
    - 不再是文绉绉报告腔
    - 改成更像高强度冲浪网友的表达
    - 但不能污染证据层
  4.5 Stage E: 接茬探针
    - 不是每次都必须开
    - 但如果能接得自然，说明它真的吃到史了

5. 这个 Agent 现在准备吃哪些项目内概念
  5.1 已有稳定观察维度

  5.3 本地搬史文化概念

6. 输出为什么要分层
  6.1 结构化证据层
    - 给程序和审阅用
    - 尽量保持中性、可追溯
  6.2 文化解释层
    - 说明“为什么这是史”
    - 解释荒诞机制和传播机制
  6.3 口吻层
    - 生成更像网友的评论
    - 但不能覆盖证据结论
  6.4 接茬层
    - 测这个 Agent 能不能顺着这坨史往下接
    - 这是能力测试，不是事实层

7. 口吻现在怎么定义
  7.1 目标不是正常分析师
    - 不是正经报告腔
    - 不是健康向微博运营号
  7.2 目标更接近
    - 高强度中文互联网用户
    - 梗密度高
    - 能懂抽象梗、政治笑话、平台黑话
    - 会区分好史、坏史、工业废史
  7.3 但仍有限制
    - 不能编媒体内容
    - 不能把不确定说成确定
    - 不能为了口嗨把分析做坏

8. 为什么它必须做成热插拔件
  8.1 因为它不是系统本体
    - 它是第一个深度分析插件
    - 以后还会有别的专项 Agent
  8.2 所以它不能反向污染 substrate
    - 不改 canonical truth
    - 不绑 CLI
    - 不绑 NapCat
    - 不强依赖 chunk 固定形式

9. 这个 Agent 怎么评估成没成
  9.1 基础能力
    - 能不能说清这是什么史
    - 能不能说清为什么是史
    - 能不能区分证据与推断
  9.2 亚文化能力
    - 能不能正确用本地史学词汇
    - 能不能看出运史官 / 二手套娃 / 工业流水线
  9.3 口吻能力
    - 能不能不再文绉绉
    - 能不能说得像真的懂这圈的人
  9.4 接茬能力
    - 能不能接一句不尬的下话茬
    - 能不能顺着史的方向接，而不是乱刷梗
