# process_AGENTs.md

> Last updated: 2026-03-10
> Scope: QQ chat preprocessing, canonical ingest, local indexing, retrieval, analyzer-facing service boundaries, and handoff into first-phase LLM analysis.

## Purpose

This file defines the preprocessing subsystem that sits after chat export and before any future analyzer, GUI, or online RAG layer.

The preprocessing subsystem is responsible for:

- ingesting exported chat data from multiple formats
- converting all inputs into one canonical message model
- building local relational and vector indexes
- performing local hybrid retrieval over those indexes
- building grounded context blocks for later analysis
- preserving image references and preparing image vectors
- projecting identity into raw and alias views
- feeding a later target-driven analysis substrate
- feeding a later dedicated LLM analysis layer

It is not responsible for:

- reranker inference
- OCR execution
- image captioning
- GUI or terminal query UX
- agent-specific conclusions such as future benshi judgments

## Phase Contract

Current phase goal:

- offline preprocessing plus local retrieval foundation

Deliverables in this phase:

- import adapters
- canonical record layer
- SQLite schema
- FTS5 text index
- local contiguous vector index
- pluggable chunk policies
- pluggable embedding and image-feature interfaces
- dual-track identity projection
- hybrid keyword/vector retrieval
- chunk-aware or window-aware context construction
- optional script-level DeepSeek generation
- stable handoff into a separate analysis substrate

Deferred to later:

- reranker choice and inference
- OCR and multimodal image parsing
- GUI and CLI adapters for retrieval
- richer answer schemas and evaluation tooling
- specialized analysis agents such as `BenshiAgent`
- first-phase open-ended LLM report design itself

## Input Sources

The preprocessing subsystem must formally support these inputs:

- exporter JSONL plus companion manifest when available
- QCE JSON
- QQ TXT transcript exports

Input fidelity levels:

- exporter JSONL: `high`
- QCE JSON: `compat`
- QQ TXT: `lossy`

All inputs must be normalized into one canonical message layer before storage or indexing.

## Canonical Truth

The canonical truth source is the message record.

Message records must remain independent from chunking.

Chunks are derived views only:

- they may be absent
- they may be generated by one or more policies
- they may change across runs
- they must never become the only source of message truth

## Storage And Indexing

Default local storage layout:

- SQLite database under `state/preprocess/db/`
- contiguous vector store under `state/preprocess/qdrant/`
- run logs and artifacts under `state/preprocess/runs/`

SQLite owns:

- import runs
- chats
- raw participants
- alias participants
- messages
- message assets
- chunk sets
- chunks
- chunk memberships
- artifacts

The current local vector store owns at least:

- `text_units`
- `image_assets`

Current phase recommendation:

- message-level text vectors are first-class
- chunk vectors are optional and policy-dependent
- image assets are vectorized independently from OCR/caption pipelines
- retrieval combines FTS5 and the local vector store rather than depending on only one index

Current local vector-storage rule:

- do not use small random-write-heavy vector backends on slow HDDs as the default local path
- prefer append-only contiguous vector files plus bounded in-memory preload for search
- keep preload conservative; do not consume all system RAM just because it is available
- when a faster SSD path exists, the vector store may be redirected there explicitly

## Chunk Policy Rules

Chunking must be policy-driven and versioned.

Required interfaces:

- `ChunkPolicy`
- `NoChunkPolicy`
- `WindowChunkPolicy`
- `TimeGapChunkPolicy`
- `HybridChunkPolicy`

Required metadata per run:

- `chunk_policy_name`
- `chunk_policy_version`
- `chunk_policy_params`

Hard rule:

- do not hard-code a repository-wide mandatory chunk size
- do not make downstream interfaces assume chunk existence

Recommended defaults may exist in code or config, but they are not architectural truths.

## Analysis Handoff Rules

The preprocessing and retrieval stack now feeds a separate analysis layer.

Important boundary:

- `qq_data_process` owns ingest, storage, indexing, retrieval, and generic context building
- `qq_data_analysis` owns target/time-scope analysis, agent orchestration, and higher-level findings
- a dedicated LLM analysis layer consumes prepared analysis packs rather than raw preprocessing internals

RAG role in the current architecture:

- it remains available as a first-class retrieval subsystem
- but its primary product role is now internal evidence grounding for agents
- it is not the default end-user interaction model

Default analysis shape expected downstream:

- choose target group/friend
- choose explicit or adaptive time scope
- internally retrieve evidence and context
- let analysis agents generate final reports

Current LLM handoff shape expected downstream:

- choose target group/friend
- choose explicit or adaptive time scope
- compress the selected window into an `analysis pack`
- let the LLM produce a high-level long report first
- only later derive tighter schemas and more specialized agents from repeated reviewed outputs

## Embedding And Image Features

Current default embedding target:

- `Qwen/Qwen3-VL-Embedding-2B`

Alternate embedding targets kept available:

- `jinaai/jina-embeddings-v4`
- OpenRouter remote embedding models

The subsystem must treat embedding as a provider interface rather than a hard-coded library call.

Required separation:

- text embedding provider
- image embedding provider
- image feature provider

Current image policy:

- keep `file_name`, `path`, `md5`, and cheap metadata
- large-scale preprocessing may skip image vectors and keep references only
- mark future multimodal parse as pending capability

Current image retrieval limitation:

- if a real image file is unavailable locally, image embedding may fall back to a text hint
- that fallback is an indexing placeholder, not final image semantics

External model/API access rule for this subsystem:

- external model/data access must use the local proxy `http://127.0.0.1:7897`
- override env var: `QQ_DATA_EXTERNAL_PROXY`
- remote embedding APIs such as OpenRouter must also follow this proxy rule
- Hugging Face cache and logs should stay inside repository-managed state paths, not the user profile
- local services such as NapCat on `127.0.0.1` must still bypass proxy
- current exception: DeepSeek analysis calls are configured for direct default networking via `state/config/llm.local.json`

Do not do in this phase:

- OCR
- caption
- VLM inference

Those belong to a later multimodal processing stage.

## Identity Projection

Privacy uses a dual-track model:

- raw identity layer
- alias identity layer

Alias requirements:

- globally stable within the local repository
- reproducible across imports
- independent from run IDs

Default outward behavior:

- analyzer inputs
- future LLM contexts
- shared reports

must use alias identity unless a dangerous override is explicitly requested.

Required dangerous option:

- `danger_allow_raw_identity_output`

This flag must live in core policy/config, not only in CLI code.

## Reserved LLM Configuration

LLM is not active in this phase, but the reserved config must be documented now:

- model: `deepseek-reasoner`
- base URL: `https://api.deepseek.com`
- API key env var: `DEEPSEEK_API_KEY`
- local secret file path used by the current analysis script:
  - `state/config/llm.local.json`

Current bounded LLM-analysis rule:

- never send the entire chat history by default
- whole-window analysis is allowed only after compression into a bounded analysis pack
- the first LLM phase is report-first, not schema-first
- first cut a dense high-signal slice from a candidate event or adaptive window
- initial target size is a few hundred messages, then shrink further if the estimated prompt-token budget would be exceeded
- print the slice plan and estimated token budget before the request
- record API `usage` after the request and save it with the analysis outputs

Do not build DeepSeek-coupled preprocessing logic yet.

The future analyzer layer must consume preprocessing and retrieval outputs through stable service interfaces.

First-stage analyzer constraints to preserve in these interfaces:

- alias identity remains the default outward projection
- findings must be able to attach evidence references
- image references should remain available even before OCR/caption exists
- nested forwards and other bizarre content patterns must stay observable in canonical data

## Decoupling Rules

The preprocessing subsystem must not import:

- `qq_data_cli`
- prompt-toolkit
- Typer
- terminal UI helpers

Preferred dependency direction:

- integrations -> exporter core -> preprocessing/retrieval core -> future analyzer/UI adapters

Never reverse that direction.

## Implementation Notes

When implementation needs new packages:

- install them from `.venv`
- use `uv add`

Keep heavyweight model runtimes out of the default runtime path until the real feature phase begins.
